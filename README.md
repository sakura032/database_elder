# 居民社区养老服务需求分析与智能匹配系统

## 1. 项目名称

居民社区养老服务系统。

## 2. 项目简介

本项目定位为面向社区养老场景的服务需求受理、机构分派、过程监管和服务评价系统。系统围绕老年人、服务需求、服务记录、服务人员、服务机构、社区六个核心实体，把“老人提交需求 -> 社区分派机构 -> 机构安排人员并完成服务 -> 老人评价”的线下养老服务过程搬到线上管理。

系统不是单纯的信息展示页，而是一个三端协同工作台：

- 老人端负责提交服务需求、维护个人资料、查看进度、查看记录和评价服务。
- 机构端负责维护服务人员、承接服务记录、安排具体人员、补全服务过程并标记完成。
- 社区端负责监管全局数据、按优先级分派需求、查看服务记录和生成运营统计。

其中 `service_demand.emergency_level` 用于区分普通、较急、紧急需求。该字段会影响社区端待分派服务排序、机构端未完成服务排序、AI 推荐机构依据、自然语言查询结果顺序和统计报告风险提示。

## 3. 账号规则

老人账号可在登录页自助注册，注册时只填写用户名和密码，系统自动创建老人档案并绑定账号。社区管理员账号与机构账号需要在 `user_account` 表中按业务主体手动准备，分别绑定对应管理主体。

## 4. 项目功能

系统按角色划分为老人、机构、社区管理员三类用户。

公共功能：

- 登录
- 登出
- 老人自助注册，注册时只填写用户名和密码，系统自动创建老人档案并绑定账号
- 修改密码
- 查看当前用户信息

老人功能：

- 查看个人基本信息
- 维护年龄、健康状态、居住地址、联系方式和需求标签
- 提交服务需求，并选择或由 AI 识别紧急程度
- 查看个人服务需求进度，紧急服务会以醒目标识展示
- 查看个人服务记录
- 对已完成服务新增评价或修改已有评价

机构端功能：

- 查看本机构信息
- 管理本机构服务人员
- 查看本机构服务过程记录，未完成记录按紧急程度优先展示
- 为服务记录安排服务人员
- 补全服务记录并标记完成
- 查看本机构服务统计

社区管理员功能：

- 管理老人信息
- 管理机构信息
- 管理服务人员信息
- 按紧急程度优先分派服务需求给机构
- 监管服务需求与服务过程记录
- 查看系统统计报表和高优先级需求风险提示

## 6. 目录结构

```text

    web/                       # 接口层，负责接收请求、检查登录状态和角色权限

      session.py               # 公共认证接口：登录、登出、注册、修改密码、当前用户信息

      org.py                   # 机构端接口：机构信息、服务人员、服务过程记录处理

    models/                    # 数据访问层，负责写 SQL 字符串并通过 PyMySQL 执行
      ai.py                    # AI 所需的需求、机构和统计上下文查询
      session.py               # 账号表 user_account 相关 SQL
      admin.py                 # 社区管理员相关 SQL
      org.py                   # 机构端相关 SQL
      elder.py                 # 老人相关 SQL
    validate/                  # 参数校验层，使用 WTForms 校验请求参数是否合法
      ai.py                    # AI 接口参数校验
      session.py               # 登录、注册、修改密码参数校验
      admin.py                 # 社区管理员查询参数校验
      org.py                   # 机构端参数校验
      elder.py                 # 老人查询参数校验
    ai/                        # DeepSeek 大模型能力封装层
      client.py                # 使用 Python 标准库 urllib 直连 DeepSeek 接口
      prompts.py               # AI 提示词模板
      service.py               # AI 业务编排和结果校正
    static/                    # 前端静态资源目录
      index.html               # Vue 页面入口
      main.js                  # Vue、Axios、ECharts 前端逻辑
      style.css                # 页面样式

  README.md                    # 项目说明文档，记录功能、结构、运行方式和注意事项
  requirements.txt             # Python 依赖列表，只安装到项目虚拟环境
  run.py                       # 项目启动入口
```

## 7. 后端结构

后端采用三层拆分：

| 目录 | 职责 |
|---|---|
| `web/` | 负责接口路由、登录状态、角色权限 |
| `models/` | 直接使用 PyMySQL 写 SQL 字符串并访问数据库 |
| `validate/` | 使用 WTForms 校验请求参数是否合法 |
| `static/` | 放置 Vue、Axios、ECharts 前端页面 |

## 11. 本地运行

### 1. 创建并激活虚拟环境

Windows PowerShell：

```powershell
py -3.10 -m venv .venv
.\.venv\Scripts\Activate.ps1
```

### 2. 安装依赖

```powershell
pip install -r requirements.txt
```

### 3. 准备数据库

远程数据库已经准备好，目标数据库为 `pension1_service`，当前核心业务表包括：

```text
community
elderly
service_demand
service_org
service_staff
service_record
staff_record_relation
user_account
```

其中 `user_account` 是认证权限表。系统不内置默认账号，管理员账号和机构账号需要按实际数据手动创建；老人账号可通过登录页自助注册。

### 4. 配置数据库连接

默认数据库配置位于 `app/config.py`：

```text
DB_HOST = 192.168.70.82
DB_PORT = 3306
DB_USER = root
DB_PASSWORD = root
DB_NAME = pension1_service
```

### 5. 配置 DeepSeek API Key

项目从 `app/config.py` 读取 DeepSeek API Key。
可选配置仍可通过环境变量覆盖：

```powershell
$env:DEEPSEEK_BASE_URL="https://api.deepseek.com"
$env:DEEPSEEK_MODEL="deepseek-v4-flash"
```

### 6. 启动项目

```powershell
python run.py
```

访问：

```text
http://127.0.0.1:5000/
```

## 12. 前端运行

当前前端放在 Flask 的 `app/static/` 目录中，不需要单独启动 Vue 开发服务器。

启动 Flask 后直接访问：

```text
http://127.0.0.1:5000/
```

前端四个页面：

| 页面 | 访问对象 | 说明 |
|---|---|---|
| 登录页 | 全部用户 | 登录、登出、查看当前账号 |
| 老人端页面 | `elder` | 提交需求、查看进度、查看记录、填写评价 |
| 机构端页面 | `org` | 管理服务人员、安排人员、补全服务记录 |
| 社区端页面 | `community_admin` | 分派需求、查看全量数据、监管统计 |

登录后页面右下角提供统一 AI 聊天框：

- 聊天框右侧快捷指令只会把示例内容填入输入框，用户确认或修改后再手动发送。
- 老人端可输入自然语言，例如“我明天想去医院复查”，AI 会识别服务类型、紧急程度和描述，并回填提交需求表单。
- 老人端在 AI 回填需求后，会生成一个后端托管的待执行操作；点击“确认执行”或回复“确认执行 / 同意 / 提交”后，系统才会真正提交服务需求。
- 如果老人继续补充“我很着急”“不是这个”“改成紧急”等内容，AI 会自动取消上一条待确认操作，重新识别并生成新的待确认操作。
- 老人端可输入“有哪些服务记录”，AI 会按当前老人账号查询本人需求和服务记录。
- 社区端可输入“帮我推荐需求 100 的承接机构”，AI 会基于需求、老人信息、紧急程度、机构服务能力和当前负载推荐机构；紧急或较急需求会优先考虑空闲人员更多、未完成记录更少的机构。
- 社区端在 AI 推荐机构后，可在聊天框中二次确认；确认时后端会再次校验权限和参数，然后分派需求并写入服务记录。
- 社区端可输入“生成统计报告”，AI 会基于数据库统计数据生成运营分析报告。
- 社区端可输入“查看待分派需求”“查服务人员”“查空闲服务人员”，AI 会把自然语言转换为安全查询意图并展示查询结果。
- 机构端可输入“推荐当前服务记录的服务人员”，AI 会基于当前选中的服务记录和本机构人员状态推荐人员。
- 机构端点击“AI 推荐”或“AI 整理”前必须先在服务记录列表点“选择”；未选择待服务老人时页面会弹窗提示，不会直接调用 AI。
- 机构端可输入“整理服务记录草稿”，AI 会生成服务类型、服务时间、服务时长和记录摘要，并回填完成记录表单；完成记录前必须选择服务人员，后端也会校验该人员属于当前机构。
- 机构端安排服务人员、完成服务记录都需要在聊天框中明确确认，确认接口二次校验通过后才会写入数据库。
- 如果用户只是询问“怎么操作 / 下一步怎么办”，AI 会作为客服助手给出操作指引，不会自动修改业务数据。

## 14. 运行注意事项

- 公共注册只允许老人账号，注册时系统自动创建并绑定老人档案；社区管理员和机构账号由数据库管理员按实际社区、机构主体预置。
- 当前项目默认连接远程数据库 `192.168.70.82 / pension1_service`；运行环境需要保证该数据库可访问。
- `user_account.role` 必须与当前远程库一致：社区端为 `community_admin`，机构端为 `org`，老人端为 `elder`；`/admin/...` 只是社区端接口路径前缀。
- 远程库已有账号可能使用 bcrypt 哈希，新注册和改密使用 Werkzeug PBKDF2 哈希，后端已同时兼容两类密码哈希。
- 接口命名遵循 `/角色/对象/动作`，例如 `/admin/elder/select`。
- 认证公共接口统一放在 `/auth/...`。
- 文档中的接口路径、角色值和数据库连接信息与当前项目实现保持一致。
